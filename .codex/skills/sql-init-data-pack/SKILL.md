# Purpose

用于规范数据库建表和初始化数据交付，保证系统不仅“有表”，而且“有足够可演示的数据”。

## Important Baseline

1. 当前仓库已经存在商品、库存、采购、销售、出入库、用户、角色、菜单等相关表。
2. 需求文档中给出的 `sys_user`、`commodity`、`inventory`、`orders`、`wave_order`、`operation_log` 等，是目标态设计方向，不等于仓库现状。
3. 本 skill 的重点是“现有表映射 + 最小侵入式扩展 + 可演示初始化数据”，不是机械重建整套库表。

## When To Use

适用于：

1. 新增业务表
2. 扩展现有商品或库存相关表
3. 生成初始化 SQL
4. 生成演示数据 SQL
5. 为截图和答辩补充测试数据

## Schema Requirements

必须先判断：

- 现有表是否可复用
- 哪些字段可扩展
- 哪些目标态能力必须新增表

如果引用目标态表名，必须明确说明它是目标设计，不是现状事实。

## Seed Data Requirements

初始化数据至少应支撑：

### 1. User

- 管理员账号
- 作业员账号

### 2. Commodity / 商品主数据

- 足够支撑列表展示
- 包含条码、名称、分类、库位等关键字段

### 3. Inventory / 库存数据

- 有正常库存
- 有低库存
- 有缺货或预警场景

### 4. Orders / 订单数据

- 支撑待处理、处理中、已完成等状态演示

### 5. Wave / 拣货任务数据

- 支撑波次或任务列表、详情页展示

### 6. Operation Logs / 日志数据

- 支撑登录、入库、订单、拣货等日志演示

如果当前阶段某些表尚未落地，必须如实说明哪些数据暂时无法初始化。

## Screenshot Support Rules

初始化数据应尽量覆盖：

- 登录页
- 商品管理
- 库存查询
- 预警列表
- 补货建议
- 订单管理
- 波次或任务管理
- 移动端作业页
- 日志页
- 用户页
- 首页看板

## Delivery Requirements

每次 SQL 交付都必须附带：

### 1. Task Goal

本次 SQL 目标。

### 2. Schema Changes

涉及哪些表、字段、索引、约束变化。

### 3. Seed Data Plan

准备了哪些业务场景数据。

### 4. Screenshot Mapping

这些数据分别支撑哪些页面或截图。

### 5. Validation

如何验证导入顺序、关联关系、数据量和状态覆盖。

### 6. Risks

与现有数据库的兼容性风险。

## Do Not

- 不要把目标态表名写成“当前仓库已有”
- 不要只给空表不给演示数据
- 不要只给极少量样例数据
- 不要忽略低库存、缺货、预警等场景
- 不要生成与现有系统结构完全脱节的数据脚本
