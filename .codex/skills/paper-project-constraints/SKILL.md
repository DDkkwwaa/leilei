# Purpose

用于约束所有 agent 的开发边界，确保项目始终符合“毕业设计配套演示系统”的定位，不跑偏成企业级复杂系统。

## Project Positioning

本项目是毕业论文配套演示系统。

项目目标：

- 轻量化
- 主链路完整
- 页面规范
- 数据可演示
- 便于截图与答辩展示

不是目标的内容：

- 高并发
- 微服务拆分
- 分布式事务
- MQ 驱动架构
- 复杂调度平台
- 复杂算法模型
- 超大规模重构

## Current Repo Baseline

约束判断时，必须先接受以下事实：

- 当前后端采用 `Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO` 技术栈
- 当前前端采用 `Vue2 + Element UI`
- 当前项目是前后端分离，但整体仍是单体应用形态
- 当前仓库是商品进销存系统，不是已经完成的前置仓系统

## Mandatory Constraints

### 1. 技术约束

- 后端沿用 `Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO`
- 前端沿用 `Vue2 + Element UI`
- 移动端目标形态为 H5，但不能默认当前仓库已经具备
- 数据库使用 `MySQL 8`
- 构建工具使用 `Maven`
- 定时任务只能使用 `Spring @Scheduled` 级别的轻量方案

### 2. 架构约束

- 必须保留单体应用
- 必须保留前后端分离结构
- 不引入微服务
- 不引入 MQ、XXL-Job、复杂调度平台
- 不为风格统一强行迁移到 MyBatis-Plus

### 3. 业务约束

项目围绕前置仓主链路建设，目标链路为：

`创建订单 -> 生成波次 -> 推送拣货 -> 扫码确认 -> 库存扣减 -> 状态更新 -> 日志记录`

但执行时必须承认，这条链路是目标态，不是当前仓库现状。

### 4. 算法约束

- 补货建议使用简单、可解释的规则
- 销量预测只允许毕业设计级别的轻量公式，例如 7 日均值思路
- 拣货路径只允许按库位编号等简单排序方式处理
- 不引入复杂优化算法、机器学习模型、图算法平台

### 5. 展示约束

- 页面必须适合论文截图
- 字段展示必须清楚
- 风格必须统一
- 数据必须可演示
- 不允许只有接口没有可展示页面

## Enforcement Rules

在规划、开发、评审时，持续检查：

1. 是否引入了不必要的复杂中间件
2. 是否做了超范围重构
3. 是否违背了当前仓库既有技术栈
4. 是否偏离了轻量化、可演示、可截图定位

## Required Check Output

如果使用本 skill 做边界检查，输出至少包含：

1. 当前任务
2. 是否符合论文项目边界
3. 偏离点
4. 修正建议

## Do Not

- 不要允许 agent 自由引入新架构
- 不要为了“更先进”而增加复杂度
- 不要把目标态能力当成仓库现状
- 不要把项目做成不利于演示和答辩的复杂系统
