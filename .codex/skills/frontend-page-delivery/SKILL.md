# Purpose
用于规范后台前端页面交付，保证每次页面开发都能说明“页面在哪里、用了什么接口、怎么测、能不能截图”。

## When To Use
适用于：
1. 商品管理页面改造
2. 库存查询与预警页面开发
3. 补货建议页面开发
4. 订单管理页面开发
5. 波次管理与详情页面开发
6. 用户管理、日志、首页看板开发
7. 菜单和路由改动后

## Delivery Requirements
每次页面交付都必须附带以下内容：

### 1. Task Goal
本次页面目标是什么。

### 2. Files To Modify
列出本次新增或修改的：
- views
- components
- router
- store
- api
- enums / constants
- permission files

### 3. Route and Menu Impact
说明：
- 页面路由
- 菜单入口
- 角色可见性
- 权限点

### 4. API Binding
说明本次页面调用哪些接口：
- 列表接口
- 详情接口
- 保存接口
- 删除接口
- 导入导出接口

### 5. UI Coverage
说明本次页面覆盖了哪些要求：
- 查询条件
- 表格列
- 状态标签
- 按钮
- 弹窗
- 详情区
- 高亮预警
- 图表展示

### 6. Self Test Path
说明如何验证：
- 进入哪个菜单
- 用什么账号
- 点击哪些操作
- 预期看到什么结果

### 7. Risks
说明：
- 依赖哪个后端字段
- 哪些接口未准备好
- 哪些枚举或字典待统一

## Screenshot-Oriented Rules
页面必须考虑：
1. 表格字段清晰
2. 按钮不杂乱
3. 状态颜色区分明显
4. 页面布局适合论文截图
5. 标题、筛选区、表格区层次清楚

## Do Not
- 不要只交页面代码，不说明入口
- 不要只说“已完成”，不说明字段与接口
- 不要做与现有后台风格冲突的页面
- 不要忽略 admin/operator 菜单差异
