# Purpose
用于接手已有仓库，快速完成“扫描现状 → 判断复用点 → 识别改造点 → 输出二开入口”的过程。

## When To Use
适用于以下场景：
1. 第一次接手当前仓库
2. 更换仓库或分支后重新评估
3. 需要在已有商品管理项目上做二次开发
4. 对现有系统结构不清楚时

## Core Outcome
输出一份可直接指导后续 agent 开发的仓库接管报告。

## Mandatory Steps

### Step 1：扫描目录结构
识别：
- 后端目录
- 前端目录
- 移动端目录
- SQL 目录
- 文档目录
- 配置文件位置

### Step 2：识别技术栈
识别：
- 后端框架
- ORM / 持久层
- 前端框架
- UI 组件库
- 构建工具
- 权限和登录机制

### Step 3：查找现有业务能力
重点查找：
- 商品管理
- 用户管理
- 菜单与路由
- 权限控制
- 操作日志
- Excel 导入导出
- 图表能力
- 是否已有移动端或 H5 支持

### Step 4：识别数据库接入方式
查找：
- SQL 初始化脚本
- 实体类
- Mapper
- 数据源配置
- 主键策略
- 表命名规则

### Step 5：输出差距分析
分成三类：
- 可复用
- 需改造
- 需新增

### Step 6：给出第一批改动建议
要明确到目录或文件层级。

## Required Output Structure
输出必须包含以下部分：
1. 仓库结构概览
2. 技术栈识别
3. 可复用模块
4. 需改造模块
5. 需新增模块
6. 数据层发现
7. 高风险点
8. 推荐第一批改动入口

## Quality Bar
一份合格的接管报告必须满足：
1. 基于代码事实
2. 能让后端 agent 直接开工
3. 能让前端 agent 找到复用点
4. 能让总控 agent 排阶段计划
5. 不停留在泛泛建议层面

## Do Not
- 不要在没扫描代码时凭经验猜测项目结构
- 不要只说技术名词，不说目录和模块位置
- 不要忽略已有商品管理模块的实际实现
- 不要直接建议重建项目
