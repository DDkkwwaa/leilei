# Purpose

用于接手已有仓库，快速完成“扫描现状 -> 判断复用点 -> 识别改造点 -> 输出二开入口”的过程。

## Important Baseline

使用本 skill 时，必须遵守：

1. 结论只能基于仓库事实。
2. 需求文档中的前置仓能力属于目标态，不等于当前仓库现状。
3. 必须明确写出“已确认存在”“缺失或未确认”“建议新增”三类内容。

## When To Use

适用于以下场景：

1. 第一次接手当前仓库
2. 仓库结构较大，需要快速识别可复用能力
3. 需要给后续 agent 输出接管报告
4. 需要判断项目适不适合做最小侵入式二开

## Mandatory Steps

### Step 1: 扫描目录结构

识别：

- 后端目录
- 前端目录
- 是否存在移动端目录
- SQL 脚本目录
- 配置文件位置
- 文档位置

### Step 2: 识别技术栈

重点确认：

- 后端框架
- 数据访问层方式
- 前端框架
- UI 组件库
- 构建工具
- 登录与权限机制

### Step 3: 识别现有业务能力

必须基于代码查找以下能力是否存在：

- 商品管理
- 用户管理
- 登录
- 菜单与路由
- 权限控制
- 日志能力
- 导入导出
- 图表能力
- 移动端或 H5 能力

### Step 4: 识别数据库接入方式

查找：

- SQL 初始化脚本
- 实体类
- Mapper / DAO
- Mapper XML
- 数据源配置
- 表结构命名特征

### Step 5: 输出差距分析

必须拆成三类：

- 可复用模块
- 需改造模块
- 需新增模块

并额外列出：

- 缺失或未确认能力
- 高风险点

### Step 6: 给出第一批改动建议

建议必须明确到目录或文件层级，不允许只写抽象模块名。

## Required Output Structure

输出必须包含以下部分：

1. 仓库结构概览
2. 技术栈识别
3. 当前已确认存在的能力
4. 缺失或未确认的能力
5. 可复用模块
6. 需改造模块
7. 需新增模块
8. 数据层发现
9. 高风险点
10. 推荐第一批改动目录和文件

## Quality Bar

一份合格的接管报告必须满足：

1. 结论都能回指到代码事实
2. 后续 agent 能直接据此做规划
3. 报告中没有把目标态写成现状
4. 不会诱导开发者推倒重来

## Do Not

- 不要在没扫描代码时凭经验猜测项目结构
- 不要只写技术名词，不写目录与文件依据
- 不要忽略现有商品进销存模块的真实实现
- 不要把“缺失”与“未确认”混为一谈
- 不要直接建议重建项目
