# Purpose

用于每个阶段或功能块完成后的交付门禁检查，确保结果满足“可运行、可验证、可演示、可截图”的最低标准。

## Important Baseline

1. 门禁检查必须基于“当前阶段范围”，不能直接按最终目标态全量打分。
2. 必须区分“当前仓库原有能力”“本阶段新增能力”“后续阶段能力”。
3. 评审必须结合代码、接口、页面、数据，不允许只看其中一项。

## When To Use

适用于：

1. 一个后端模块开发完成后
2. 一个前端页面组完成后
3. 一个移动端流程完成后
4. SQL 与初始化数据完成后
5. 准备进入下一阶段前
6. 准备最终交付前

## Gate Dimensions

### 1. Backend Gate

检查：

- 是否有明确接口
- 是否符合业务规则
- 是否可被前端调用
- 是否说明数据库影响
- 是否有自测方法

### 2. Frontend Gate

检查：

- 页面是否可访问
- 菜单路由是否可达
- 接口绑定是否明确
- 字段是否完整
- 状态和按钮展示是否正确
- 是否适合截图

### 3. Mobile Gate

检查：

- 页面是否适配手机
- 流程是否闭环
- 扫码逻辑是否清楚
- 弱网处理是否有方案
- 是否适合演示

### 4. SQL Gate

检查：

- 表结构是否完整
- 字段是否与当前阶段需求一致
- 初始化数据是否足够
- 数据场景是否覆盖预警、缺货、完成态等演示
- 是否支撑截图与联调

### 5. Screenshot Gate

检查当前阶段涉及页面是否具备可截图性，例如：

- 登录页
- 看板
- 商品管理
- 库存与预警
- 补货建议
- 订单管理
- 波次或任务管理
- 移动端作业页
- 日志页
- 用户管理

如果某页面不在当前阶段范围，不能作为阻塞项。

## Required Review Output

每次做门禁检查时，输出必须包含：

1. 检查对象
2. 当前阶段范围
3. 已满足项
4. 未满足项
5. 阻塞项
6. 是否可进入下一阶段
7. 修复建议

## Gate Decision Rules

- 主链路在当前阶段范围内且被打断：不得进入下一阶段
- 页面不可访问：不能作为可交付页面
- SQL 数据不足以支撑当前阶段演示：不得进入截图或答辩准备
- 仅存在非关键样式问题：可带问题进入下一阶段
- 角色权限不符合要求：视为高优先级问题

## Do Not

- 不要只看代码不看演示效果
- 不要只看页面不看业务链路
- 不要忽略初始化数据质量
- 不要用后续阶段要求否定当前阶段交付
- 不要在主链路阻塞时给出“可交付”判断
