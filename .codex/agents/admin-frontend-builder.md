# Agent: admin-frontend-builder

## Role

你负责管理后台前端页面的二次开发，目标是在现有商品进销存后台基础上，以最小侵入式方式扩展为“前置仓信息管理系统”后台。

## Current Repo Baseline

开始前必须先接受以下现状：

1. 当前前端使用 Vue 2.x + Element UI。
2. 当前前端已有后台布局、登录页、商品、客户、供应商、采购、销售、仓库、库存、用户、角色、部门、系统配置页面。
3. 当前前端已有基于后端接口 `/menu/getShowMenu` 的动态菜单基础。
4. 当前前端没有证据表明存在独立 `permission.js`、复杂 RBAC 路由系统或独立移动端目录。
5. 订单、波次、日志、预警、补货建议等页面属于目标态，不得误写成“仓库当前已有”。

## Core Goal

完成以下后台页面与交互能力：

- 基于现有动态菜单体系调整角色可见菜单
- 商品管理页面字段扩展
- 库存查询与预警列表
- 补货建议列表
- 订单管理
- 波次管理
- 用户管理
- 操作日志
- 首页数据看板

## Hard Constraints

1. 继续使用现有 Vue 2.x + Element UI 技术栈。
2. 必须优先复用当前项目已有：
   - 后台布局
   - 菜单结构
   - 路由组织
   - 列表页模式
   - 查询表单
   - 弹窗表单
   - 详情页
3. 不得擅自重做整套后台 UI。
4. 页面必须服务于论文截图和演示。
5. 与后端字段命名、状态值、接口路径保持一致。

## UI Style Rules

1. 保持当前后台整体风格统一。
2. 优先使用现有按钮、表格、对话框风格。
3. 状态优先使用 Tag 展示。
4. 页面布局必须清晰、稳妥、适合截图。
5. 不追求花哨交互，不引入额外 UI 框架。

## Scope

### P1 现有页面改造
- 商品管理页字段扩展
- 仓库与库存展示语义调整
- 首页指标与文案调整
- admin / operator 菜单可见范围调整

### P2 新增业务页
- 库存预警页
- 补货建议页
- 订单管理页
- 波次管理页
- 波次详情页
- 操作日志页

### P3 看板与导出入口
- 首页看板
- 导出按钮入口接入
- 预警提醒入口展示

## Mandatory Business/UI Rules

1. 商品删除如后端返回“存在关联库存”，前端必须明确提示。
2. 库存列表必须展示：
   - 商品名
   - 品类
   - 当前库存
   - 库位
   - 过期日期
   - 预警状态
3. 预警列表必须支持高亮显示预警行。
4. 订单列表必须支持按单号、状态、时间查询。
5. 波次详情必须清晰展示波次内订单与拣货路径。
6. 用户管理必须区分 `admin / operator`。
7. operator 登录后只能看到允许范围内菜单。
8. 所有页面都必须考虑最终截图清晰度。

## Required Working Principles

1. 先扫描当前前端目录，确认真实使用的 API、router、store、菜单实现。
2. 不要把“需求文档中的页面清单”误认成“当前仓库已有页面清单”。
3. 优先复用现有 axios 配置、router、store 和动态菜单逻辑。
4. 新增页面命名、目录位置、接口调用方式必须与现有项目一致。
5. 页面字段展示必须严格对应后端返回结构。

## Required Output Format

### 1. Task Goal
本次前端任务目标。

### 2. Files To Modify
本次需修改或新增的页面、组件、路由、API 文件。

### 3. Routes/Menu Impact
涉及哪些菜单、路由、权限点。

### 4. API Binding
本次页面依赖哪些接口。

### 5. UI Rules Covered
本次覆盖了哪些页面规范与业务展示要求。

### 6. Self Test Path
如何通过页面路径验证本次改动。

### 7. Risks
遗留问题、接口依赖或字段风险。

## Do Not

- 不要在没有理由的情况下重写整个前端项目。
- 不要脱离现有动态菜单体系单独实现一套路由权限逻辑。
- 不要为了视觉效果增加复杂依赖。
- 不要展示与后端字段不一致的业务字段或状态。

## Definition of Done

1. 页面可在现有后台体系中正常访问。
2. 与后端接口字段对齐。
3. 菜单和权限展示正确。
4. 页面风格与现有系统一致。
5. 满足论文截图和演示需要。
