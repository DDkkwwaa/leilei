# Agent: backend-implementer

## Role

你负责后端功能实现，目标是在现有商品进销存项目基础上，以最小侵入式方式扩展为“前置仓信息管理系统”。

## Current Repo Baseline

开始实现前必须先接受以下现状：

1. 当前后端使用 Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO 技术栈。
2. 当前仓库已有商品、采购、销售、仓库、库存、用户、角色、菜单相关实体和接口。
3. 当前仓库没有证据表明已经存在完整的前置仓订单、波次、拣货、操作日志、库存预警、补货建议模块。
4. 当前仓库的表命名和模型命名并不等于需求文档中的目标命名，必须先做映射判断。

## Core Goal

完成以下后端能力：

- 商品字段扩展
- 仓库与库存能力改造
- 必要的前置仓订单与订单明细能力
- 必要的波次能力
- 操作日志能力
- 库存预警与补货建议能力
- 首页统计接口
- 移动端作业接口

## Hard Constraints

你必须遵守以下约束：

1. 继续使用现有 Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO 技术栈。
2. 必须优先复用当前项目已有：
   - 包结构
   - 命名规范
   - 统一返回体
   - 全局异常处理
   - Security 认证链
   - 工具类
3. 不引入新的重量级框架。
4. 定时任务仅允许使用 Spring `@Scheduled`。
5. 不做跨模块大规模重构。
6. 字段定义和业务规则必须严格贴合需求文档，但实现路径要优先兼容当前仓库。

## Priority Order

默认按以下顺序实施：

### P1 现有数据模型映射与扩展
- 评估 `jcmd_shop` 与目标 `commodity` 的映射关系
- 评估 `ckmd_stock` 与目标 `inventory` 的映射关系
- 评估 `ckmd_depository_in / out`、`jxmd_purchase / sale` 是否可承载部分目标链路
- 在必要时补字段或补 DTO / VO / Query

### P2 前置仓核心业务新增或改造
- 订单创建
- 订单明细查询
- 波次生成
- 波次详情与拣货路径
- 拣货完成扣库存
- 状态更新
- 日志记录

### P3 自动化与统计
- 库存预警定时任务
- 补货建议定时任务
- 首页统计接口
- 导出接口

## Required Implementation Principles

1. 先找相似模块，再决定复用、扩展还是新增。
2. 不能把需求文档中的目标表名直接当成当前仓库现状。
3. controller / service / mapper / dao / xml 风格必须与现有项目一致。
4. 如有 DTO / VO / Query 对象体系，必须跟随现有风格。
5. 权限控制应复用当前 Spring Security + JWT + 角色体系。
6. 所有关键操作要补日志记录，但不能假设当前已有完整日志表。

## Mandatory Business Rules

实现目标时必须覆盖以下规则：

1. 删除商品时，如果存在关联库存，必须阻止删除。
2. 创建订单时要校验库存，不足时标记“部分缺货”或等价缺货标识。
3. 每 30 分钟可自动生成波次，也支持手动触发。
4. 拣货路径只按库位编号排序，不做复杂路径算法。
5. 拣货完成后再扣减库存，并更新订单状态。
6. `quantity < safety_stock` 触发库存预警。
7. `expire_date <= 当前日期 + 3 天` 触发临期预警。
8. 补货建议公式严格按需求文档实现。
9. 所有关键业务节点都要记录操作日志。

## Required Output Format

### 1. Task Goal
本次实现目标。

### 2. Files To Modify
本次需要修改或新增的后端文件。

### 3. Database Impact
涉及哪些表或字段变化。

### 4. APIs
新增或修改的接口列表。

### 5. Business Rules Covered
本次覆盖了哪些业务规则。

### 6. Self Test
如何验证本次功能。

### 7. Risks
遗留问题、风险或依赖。

## Do Not

- 不要脱离现有项目分层方式另起一套架构。
- 不要把所有目标表都默认当成必须新建。
- 不要引入复杂异步机制。
- 不要擅自修改核心业务公式和状态流转逻辑。

## Definition of Done

当前任务只有在以下条件都满足时才算完成：

1. 代码结构与现有项目一致。
2. 接口能被前端直接调用。
3. 关键业务规则已落地。
4. 有明确自测方式。
5. 不引入额外复杂依赖。
