# Codex Workflow

本工作流用于指导 Codex 在当前仓库中完成“基于现有商品进销存项目，最小侵入式二次开发为前置仓信息管理系统”的全过程。

## Workflow Baseline

在进入阶段流程前，所有 agent 必须先接受以下仓库事实：

1. 当前后端是 `Spring Boot 2.7.6` 单体多模块项目。
2. 当前后端数据访问层是 `MyBatis Generator + Mapper XML + DAO`，不是 MyBatis-Plus。
3. 当前前端是 `Vue2 + Element UI` 的后台管理项目。
4. 当前仓库已具备登录、用户、角色、菜单、权限、商品、库存、采购、销售、出入库等基础能力。
5. 当前仓库尚未确认已具备完整的前置仓订单、波次、拣货、移动端 H5、操作日志中心、补货建议、成熟 Excel 导入等能力。
6. 需求文档描述的是二次开发目标态，不等于当前仓库现状。

---

## Stage 0: 仓库接管

负责人：`repo-analyst`

### 目标

- 识别当前仓库目录结构与技术栈
- 判断前后端拆分方式与数据库接入方式
- 找出现有可复用模块
- 明确缺失能力与高风险点

### 输出

- 仓库接管报告
- 可复用模块清单
- 需改造模块清单
- 需新增模块清单
- 高风险点
- 第一批改动目录与文件建议

### 通过标准

- 结论必须有代码事实支撑
- 必须区分“已有能力”和“目标能力”
- 不能凭需求文档倒推现状

---

## Stage 1: 总体规划

负责人：`delivery-orchestrator`

### 目标

- 基于接管报告与需求文档制定二开总计划
- 划分阶段边界与优先级
- 明确每阶段复用、改造、新增范围

### 输出

- 阶段划分
- 每阶段目标
- agent 分工
- 每阶段涉及目录和重点文件
- 每阶段风险与验证方式
- 第一阶段优先改动项

### 通过标准

- 不能提出推倒重来方案
- 必须保留单体 Spring Boot + Vue2 前后端分离结构
- 必须沿用 MBG + XML + DAO 体系

---

## Stage 2: 数据层与后端骨架

负责人：`db-migration-seeder` + `backend-implementer`

### 目标

- 先完成现有表结构与目标业务模型的映射
- 明确哪些表复用、哪些扩字段、哪些新增
- 落地第一批后端接口与数据库脚本

### 输出

- 数据模型映射说明
- SQL 变更脚本
- 初始化与演示数据方案
- 实体、Mapper、XML、Service、Controller 改造或新增方案

### 通过标准

- 不允许脱离现有数据层体系另起一套 ORM
- 不允许直接假定 `commodity`、`inventory`、`orders` 等目标表已存在
- 初始化数据必须能支撑页面展示和联调

---

## Stage 3: 核心业务链路

负责人：`backend-implementer` + `admin-frontend-builder` + `mobile-h5-builder`

### 目标

按阶段打通前置仓主链路，目标链路为：

`创建订单 -> 生成波次 -> 推送拣货 -> 扫码确认 -> 库存扣减 -> 状态更新 -> 日志记录`

但在执行时必须承认：

- 该链路是目标态，不是当前仓库现状
- 若某阶段后端能力未齐，前端和 H5 只能接已实现部分

### 输出

- 后端核心业务接口
- PC 管理端页面
- 移动端/H5 作业页面
- 状态流转与日志留痕能力

### 通过标准

- 至少有一条真实可跑通的主链路
- 页面、接口、数据三者一致
- 不能只做静态页面或伪流程

---

## Stage 4: 辅助能力与展示增强

负责人：`backend-implementer` + `admin-frontend-builder`

### 目标

- 库存预警
- 补货建议
- 看板展示
- 导出能力
- 适合论文截图与答辩演示的展示优化

### 输出

- 预警规则与展示页
- 补货建议接口与页面
- 首页看板或统计页
- 导出接口与按钮

### 通过标准

- 只能使用毕业设计范围内的轻量方案
- 不能引入 MQ、XXL-Job、复杂算法、复杂调度平台
- 展示能力必须依赖真实数据，不允许纯假图

---

## Stage 5: 验收与交付

负责人：`qa-reviewer` + `delivery-orchestrator`

### 目标

- 核查阶段承诺内容是否已真实交付
- 核查新能力是否破坏旧系统
- 核查是否具备演示、截图、答辩最低要求

### 输出

- QA 评审报告
- 问题清单
- 阻塞项与修复建议
- 是否允许进入下一阶段的结论

### 通过标准

- 结论必须有证据
- 必须区分“本阶段未做”与“做了但有缺陷”
- 不允许用目标态要求直接否定未纳入本阶段的事项

---

## Global Delivery Rules

1. 每次优先推进一个清晰业务块，不做散点式大改。
2. 未完成仓库接管前，不允许直接大改业务代码。
3. 未明确数据库映射关系前，不允许盲目新增整套目标表。
4. 所有改动都要遵守最小侵入式原则。
5. 所有页面、数据和流程都要服务于真实演示，不是只服务于文档描述。
6. 所有 agent 必须区分“当前仓库现状”和“前置仓目标态”。
