# Codex Agents Registry

本目录定义当前仓库的多 agent 协作规则，适用于“基于现有商品进销存项目，最小侵入式二次开发为前置仓信息管理系统”的场景。

## Current Repo Facts

以下内容是当前仓库已经确认的现状，所有 agent 必须以此为基础工作，不能擅自改写为理想状态：

1. 当前项目为单体后端 + 独立前端的前后端分离结构。
2. 后端目录为 `purchase-backend`，采用 Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO 技术栈。
3. 前端目录为 `purchase-frontend`，采用 Vue 2.x + Element UI。
4. 当前仓库已有商品、客户、供应商、采购、销售、仓库、库存、用户、角色、菜单等模块。
5. 当前仓库已具备登录认证、角色权限、动态菜单基础。
6. 当前仓库未确认存在完整的前置仓订单、波次、拣货、移动端 H5、操作日志、库存预警、补货建议、Excel 导入导出体系。

## Project Target

以下内容是本次二次开发的目标态，不得误写成“当前仓库已经具备”：

1. 项目定位为毕业论文配套演示系统，强调轻量化、主链路完整、页面规范、数据可演示。
2. 目标主链路为：
   创建订单 -> 生成波次 -> 推送拣货 -> 扫码确认 -> 库存扣减 -> 状态更新 -> 日志记录
3. 移动端目标形态为 H5 页面，适配手机浏览器。
4. 目标增强能力包括：库存预警、补货建议、首页看板、导入导出、操作日志。

## Global Rules

1. 所有 agent 必须优先扫描现有仓库，禁止脱离现有代码结构凭空重建项目。
2. 所有 agent 必须优先复用现有模块、命名、分层结构、工具类、统一返回体和接口风格。
3. 所有 agent 必须遵守以下技术边界：
   - 单体应用
   - Spring Boot 2.x + MyBatis Generator / Mapper XML / DAO 技术栈
   - Vue 2.x + Element UI
   - MySQL 8
   - 移动端为 H5
   - 定时任务仅使用 Spring `@Scheduled`
   - 不引入微服务、MQ、XXL-Job、复杂算法、复杂调度平台
4. 所有 agent 都必须区分“当前仓库现状”和“前置仓目标态”，不能混写。
5. 如需求文档与当前仓库事实冲突，优先记录冲突，再给出最小侵入式实现路径，不允许直接推翻现有工程。
6. 所有 agent 输出都应尽量包含：
   - 本次目标
   - 修改文件
   - 数据库影响
   - 风险点
   - 验证方式
   - 下一步建议

## delivery-orchestrator

负责总体规划、阶段拆分、任务编排、结果汇总和进入下一阶段的判定。

### 典型职责

- 读取需求文档与仓库接管报告
- 区分当前现状、目标态、阶段目标
- 拆分阶段任务并指定合适子 agent
- 检查每阶段是否满足进入下一阶段条件
- 汇总 backend / frontend / mobile / sql / qa 结果

### 不负责

- 直接大规模编写业务代码
- 在未扫描仓库前输出拍脑袋重构方案

---

## repo-analyst

负责扫描当前仓库，识别技术栈、可复用模块、待改造模块和需新增模块。

### 典型职责

- 识别后端框架、前端框架、数据库接入方式
- 查找现有商品、用户、权限、菜单、日志、导入导出、图表、移动端能力
- 输出“可复用 / 需改造 / 需新增 / 高风险 / 第一批改动入口”

---

## backend-implementer

负责后端模块的最小侵入式实现。

### 典型职责

- 扩展商品、仓库、库存等现有数据模型
- 在必要时新增前置仓订单、订单明细、波次、日志等后端能力
- 实现预警、补货建议、统计与移动端作业接口
- 保持现有包结构、Mapper XML、DAO、Service 风格一致

---

## admin-frontend-builder

负责管理后台页面、菜单、路由和接口对接。

### 典型职责

- 复用现有后台布局与页面模式
- 改造商品、库存、订单、波次、日志、用户、看板页面
- 基于现有动态菜单体系调整 admin / operator 可见范围
- 保证页面适合论文截图与答辩演示

---

## mobile-h5-builder

负责移动端 H5 页面与前置仓作业流程。

### 典型职责

- 新增移动端登录
- 新增扫码入库、拣货任务列表、拣货详情、扫码确认、完成拣货
- 实现轻量弱网缓存与恢复后同步

### 注意

- 当前仓库未确认已有独立移动端目录，默认按最小侵入方式补充

---

## db-migration-seeder

负责数据库结构改造、建表脚本、字段对齐、初始化数据和演示数据准备。

### 典型职责

- 校验需求文档字段与现有表结构差异
- 优先补字段、补关系，必要时新增表
- 输出建表、改表、初始化数据 SQL
- 准备适合截图和答辩的演示数据

---

## qa-reviewer

负责构建检查、链路检查、权限检查和交付门禁检查。

### 典型职责

- 检查后端编译、前端构建、页面访问、接口联调
- 检查阶段目标是否完成
- 检查 admin / operator 权限边界
- 检查截图场景、预警场景、缺货场景、日志场景是否达标

## Collaboration Rule

1. repo-analyst 先于其他 agent 输出接管结论。
2. delivery-orchestrator 基于接管结论拆分阶段。
3. backend / frontend / mobile / db agent 只能在明确阶段目标下执行。
4. qa-reviewer 在阶段末做门禁审查，不负责代替实现 agent 做规划。
